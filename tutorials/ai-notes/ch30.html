一、什么是边缘AI——为什么要在端侧部署

传统的AI部署模式是将模型运行在云端服务器上，用户通过网络请求获取推理结果。但随着物联网和移动设备的普及，这种模式面临诸多挑战：网络延迟（远程服务器往返可能需要几百毫秒）、隐私风险（敏感数据上传到云端）、带宽成本（大量数据传输带来高额费用）、离线可用性（无网络环境下无法工作）。边缘AI（Edge AI）正是为解决这些问题而生——它将模型直接部署在用户设备上进行推理，让AI能力触手可及。

边缘AI vs 云AI：核心对比

边缘AI的核心优势在于低延迟（本地推理无需网络往返，响应时间从数百毫秒降到个位数毫秒）、数据隐私（数据不出设备，天然保护用户隐私）、离线可用（无需网络连接即可工作）、降低带宽成本（减少数据传输量）。但边缘AI也面临挑战：设备算力有限、内存受限、功耗敏感，因此需要对模型进行压缩和优化。

实际案例：iPhone的Face ID就是典型的边缘AI——面部识别模型完全运行在手机的Neural Engine芯片上，人脸数据从不离开你的手机。

二、模型压缩技术

要在资源受限的设备上运行AI模型，首先需要将模型"瘦身"。模型压缩的核心目标是：在尽量保持模型精度的前提下，减少模型的存储占用和计算量。三大主流压缩技术分别是量化、剪枝和知识蒸馏。

1. 量化（Quantization）

量化是将模型中的浮点数权重（通常为FP32/FP16）转换为低精度整数表示（INT8或INT4）。例如，一个100MB的FP32模型，量化为INT8后大小约为25MB，推理速度可提升2-4倍。量化分为训练后量化（PTQ）——在训练完成后直接量化，简单快速；量化感知训练（QAT）——在训练过程中模拟量化效果，精度损失更小。

核心公式：$\text{quantized\_value} = \text{round}(\frac{\text{float\_value}}{\text{scale}}) + \text{zero\_point}$，其中scale和zero\_point是量化参数。

2. 剪枝（Pruning）

剪枝是移除模型中不重要的连接、神经元或整个层。就像修剪树枝一样，剪掉"不活跃"的部分可以减小模型体积并加速推理。常见的剪枝方式包括非结构化剪枝（移除单个权重，需要特殊硬件支持）和结构化剪枝（移除整个通道或层，对硬件更友好，是边缘部署的首选）。

3. 知识蒸馏（Knowledge Distillation）

知识蒸馏是用一个大的"教师模型"（Teacher）来训练一个小的"学生模型"（Student）。学生模型学习教师模型的输出分布（soft labels），而非仅学习硬标签（hard labels）。这样，一个只有教师模型十分之一参数量的学生模型，可以达到接近教师模型的精度。

经典案例：DistilBERT是BERT的蒸馏版本，参数量减少40%，推理速度提升60%，而精度仅下降约3%。
压缩技术 压缩比 精度损失 实现难度 适用场景
INT8量化 4× 1-3% 低 通用场景，首选方案
INT4量化 8× 3-8% 中 极端资源受限场景
结构化剪枝 2-5× 1-5% 中 需要硬件友好压缩
知识蒸馏 5-10× 2-5% 高 训练新小模型
实战建议

在实际部署中，量化是最常用且性价比最高的压缩方式。建议先尝试INT8量化，如果精度损失不可接受，再考虑QAT或结合剪枝。INT4量化虽然压缩比更高，但对模型精度影响较大，通常用于超大模型（如LLM）的端侧部署。

代码示例：模型量化INT8

import torch
import torch.quantization as quant
import torchvision.models as models

# 加载预训练的MobileNetV2
model = models.mobilenet_v2(pretrained=True)
model.eval()

# 方法一：动态量化（适用于线性层为主的模型）
quantized_model = torch.quantization.quantize_dynamic(
    model,
    {torch.nn.Linear},  # 对Linear层进行量化
    dtype=torch.qint8
)

# 查看量化前后的模型大小对比
import os
torch.save(model.state_dict(), '/tmp/full_model.pth')
torch.save(quantized_model.state_dict(), '/tmp/quant_model.pth')
print(f"原始模型: {os.path.getsize('/tmp/full_model.pth') / 1e6:.2f} MB")
print(f"量化模型: {os.path.getsize('/tmp/quant_model.pth') / 1e6:.2f} MB")

# 方法二：静态量化（精度更高，需要校准数据）
model_fp32 = models.mobilenet_v2(prepared=True)
model_fp32.eval()

# 配置量化方案
model_fp32.qconfig = quant.get_default_qconfig('fbgemm')

# 准备量化
model_prepared = quant.prepare(model_fp32)

# 使用校准数据集（这里用随机数据演示）
calibration_data = [torch.randn(1, 3, 224, 224) for _ in range(100)]
with torch.no_grad():
    for data in calibration_data:
        model_prepared(data)

# 执行量化
model_int8 = quant.convert(model_prepared)
print("INT8静态量化完成！")

# 推理对比
dummy_input = torch.randn(1, 3, 224, 224)
with torch.no_grad():
    output_fp32 = model(dummy_input)
    output_int8 = model_int8(dummy_input)
    print(f"输出差异: {(output_fp32 - output_int8).abs().mean():.6f}")
量化注意事项

量化后务必在实际目标设备上验证精度。某些对数值敏感的模型（如医学影像分割）可能在INT8下精度损失较大，此时应优先考虑FP16半精度或QAT量化感知训练。量化后的模型文件扩展名通常为.onnx或.tflite，需使用对应推理引擎加载。

三、ONNX Runtime——跨平台推理引擎

ONNX（Open Neural Network Exchange）是微软推出的开放模型格式，旨在实现不同深度学习框架之间的模型互操作性。ONNX Runtime是其官方推理引擎，支持Windows、Linux、macOS、Android、iOS等多个平台，并能自动利用CPU（AVX/AVX2/AVX-512）、GPU（CUDA/DirectML）、NPU等硬件加速。它已成为边缘AI部署的事实标准之一。

ONNX工作流

标准的ONNX部署流程：PyTorch/TensorFlow训练 → 导出为ONNX格式 → ONNX Runtime推理。这个流程的最大优势是"一次导出，多处运行"——同一个ONNX模型可以在服务器、手机、嵌入式设备上运行，无需针对每个框架单独优化。

为什么选择ONNX Runtime？相比TensorFlow Lite和Core ML各自锁定生态，ONNX Runtime提供统一的跨平台方案，支持超过30种硬件加速器后端。

代码示例：ONNX导出与优化

import torch
import torch.onnx
import onnxruntime as ort
import numpy as np

# ========== 步骤1：训练或加载模型 ==========
model = torch.hub.load('pytorch/vision', 'resnet18', pretrained=True)
model.eval()

# ========== 步骤2：导出为ONNX格式 ==========
dummy_input = torch.randn(1, 3, 224, 224)
torch.onnx.export(
    model,
    dummy_input,
    "resnet18.onnx",
    export_params=True,
    opset_version=13,           # ONNX算子集版本
    do_constant_folding=True,   # 常量折叠优化
    input_names=['input'],
    output_names=['output'],
    dynamic_axes={              # 支持动态batch size
        'input': {0: 'batch_size'},
        'output': {0: 'batch_size'}
    }
)
print("ONNX模型导出成功！")

# ========== 步骤3：使用ONNX Runtime推理 ==========
# CPU推理
sess_cpu = ort.InferenceSession(
    "resnet18.onnx",
    providers=['CPUExecutionProvider']
)

# GPU推理（如果有CUDA）
providers = ['CUDAExecutionProvider', 'CPUExecutionProvider']
sess_gpu = ort.InferenceSession("resnet18.onnx", providers=providers)

# 执行推理
input_data = np.random.randn(1, 3, 224, 224).astype(np.float32)
outputs_cpu = sess_cpu.run(None, {'input': input_data})
outputs_gpu = sess_gpu.run(None, {'input': input_data})

print(f"CPU输出shape: {outputs_cpu[0].shape}")
print(f"GPU输出shape: {outputs_gpu[0].shape}")

# ========== 步骤4：ONNX模型优化 ==========
from onnxruntime.transformers import optimizer

# 自动优化模型（融合算子、消除冗余等）
optimized_model = optimizer.optimize_model(
    "resnet18.onnx",
    model_type='bert',
    num_heads=12,
    hidden_size=768
)
optimized_model.save_model_to_file("resnet18_optimized.onnx")
print("ONNX模型优化完成！")
ONNX Runtime性能调优技巧

1. 启用图优化：设置 SessionOptions graph_optimization_level 为 ORT_ENABLE_ALL；2. 设置线程数：intra_op_num_threads 匹配设备CPU核心数；3. 使用量化：ONNX Runtime内置了动态量化和静态量化工具，可在导出时直接应用。

四、TensorRT——NVIDIA GPU优化

TensorRT是NVIDIA推出的高性能深度学习推理优化器和运行时，专为NVIDIA GPU设计。它通过层融合（将多个算子合并为一个）、精度校准（自动FP16/INT8量化）、内核自动调优（为特定GPU选择最优计算内核）等技术，可以将推理速度提升2-5倍。TensorRT广泛应用于自动驾驶（NVIDIA DRIVE）、数据中心推理、机器人等场景。

TensorRT优化原理

  • 层融合（Layer Fusion）：将Conv + BN + ReLU等连续操作合并为单个CUDA内核，减少内存读写次数
  • 内核自动调优（Auto-tuning）：为每一层搜索当前GPU上最快的CUDA内核实现
  • 动态Tensor内存：复用不同层之间的内存空间，减少显存占用
  • 多流执行：利用CUDA Stream实现推理任务的流水线并行

代码示例：TensorRT加速推理

import tensorrt as trt
import numpy as np

# TensorRT日志器
TRT_LOGGER = trt.Logger(trt.Logger.WARNING)

def build_engine(onnx_path, engine_path, precision='fp16'):
    """将ONNX模型转换为TensorRT引擎"""
    builder = trt.Builder(TRT_LOGGER)
    network = builder.create_network(
        1 << int(trt.NetworkDefinitionCreationFlag.EXPLICIT_BATCH)
    )
    parser = trt.OnnxParser(network, TRT_LOGGER)

    # 解析ONNX模型
    with open(onnx_path, 'rb') as f:
        if not parser.parse(f.read()):
            for error in range(parser.num_errors):
                print(parser.get_error(error))
            return None

    # 配置构建参数
    config = builder.create_builder_config()
    config.set_memory_pool_limit(trt.MemoryPoolType.WORKSPACE, 1 << 30)  # 1GB

    # 设置精度
    if precision == 'fp16' and builder.platform_has_fast_fp16:
        config.set_flag(trt.BuilderFlag.FP16)
        print("启用FP16精度加速")
    elif precision == 'int8' and builder.platform_has_fast_int8:
        config.set_flag(trt.BuilderFlag.INT8)
        # 需要提供校准器
        # config.int8_calibrator = MyCalibrator(calibration_data)
        print("启用INT8精度加速")

    # 构建引擎（耗时较长，建议缓存）
    engine = builder.build_serialized_network(network, config)

    # 保存引擎
    with open(engine_path, 'wb') as f:
        f.write(engine)
    print(f"TensorRT引擎已保存: {engine_path}")
    return engine

def infer_with_tensorrt(engine_path, input_data):
    """使用TensorRT引擎进行推理"""
    runtime = trt.Runtime(TRT_LOGGER)
    with open(engine_path, 'rb') as f:
        engine = runtime.deserialize_cuda_engine(f.read())
    context = engine.create_execution_context()

    # 分配GPU内存
    import pycuda.driver as cuda
    import pycuda.autoinit
    d_input = cuda.mem_alloc(input_data.nbytes)
    output = np.empty((1, 1000), dtype=np.float32)
    d_output = cuda.mem_alloc(output.nbytes)

    # 设置输入shape（动态shape场景必需）
    context.set_binding_shape(0, input_data.shape)

    # 执行推理
    stream = cuda.Stream()
    context.execute_async_v2(
        bindings=[int(d_input), int(d_output)],
        stream_handle=stream.handle
    )
    stream.synchronize()

    # 拷贝结果回CPU
    cuda.memcpy_dtoh(output, d_output)
    return output

# build_engine("resnet18.onnx", "resnet18.trt", precision='fp16')
# result = infer_with_tensorrt("resnet18.trt", np.random.randn(1,3,224,224).astype(np.float32))
TensorRT使用注意

1. TensorRT引擎与特定GPU架构绑定，在T4上构建的引擎不能直接在A100上运行（需重新构建）；2. 首次构建引擎较慢（几分钟到几十分钟），但构建结果可缓存复用；3. INT8量化需要提供校准数据集，通常需要100-1000张代表性图片。

五、移动端部署——TFLite与Core ML

移动端是边缘AI最大的应用场景。两大主流平台各有专属方案：TensorFlow Lite（TFLite）是Google推出的轻量级推理框架，支持Android/iOS/Web，兼容性极佳；Core ML是Apple的端侧推理框架，深度集成iOS/macOS生态，能自动利用Neural Engine、GPU、CPU三种硬件加速。

TensorFlow Lite
支持Android/iOS/Web/嵌入式，提供C++/Java/Python API，内置NNAPI（Android Neural Networks API）硬件加速。
适合跨平台统一部署，生态成熟
Core ML
Apple生态专属，自动利用Neural Engine（A系列/M系列芯片），支持模型转换工具coremltools。
iOS性能最优，与Swift/Objective-C无缝集成

代码示例：TFLite移动端部署

import tensorflow as tf
import numpy as np

# ========== 步骤1：转换模型为TFLite格式 ==========
# 加载预训练的MobileNetV2
model = tf.keras.applications.MobileNetV2(
    weights='imagenet',
    input_shape=(224, 224, 3)
)

# 创建TFLite转换器
converter = tf.lite.TFLiteConverter.from_keras_model(model)

# 启用优化（默认动态量化）
converter.optimizations = [tf.lite.Optimize.DEFAULT]

# 方法A：动态量化（最简单，无需校准数据）
tflite_quant_model = converter.convert()

# 保存模型
with open('mobilenetv2_quant.tflite', 'wb') as f:
    f.write(tflite_quant_model)
print(f"TFLite模型大小: {len(tflite_quant_model) / 1e6:.2f} MB")

# 方法B：全整数量化（需要代表性数据集）
def representative_dataset():
    """提供校准数据"""
    for _ in range(100):
        data = np.random.rand(1, 224, 224, 3).astype(np.float32)
        yield [data]

converter_int8 = tf.lite.TFLiteConverter.from_keras_model(model)
converter_int8.optimizations = [tf.lite.Optimize.DEFAULT]
converter_int8.representative_dataset = representative_dataset
converter_int8.target_spec.supported_ops = [
    tf.lite.OpsSet.TFLITE_BUILTINS_INT8
]
converter_int8.inference_input_type = tf.int8
converter_int8.inference_output_type = tf.int8

tflite_int8_model = converter_int8.convert()
with open('mobilenetv2_int8.tflite', 'wb') as f:
    f.write(tflite_int8_model)
print(f"INT8 TFLite模型大小: {len(tflite_int8_model) / 1e6:.2f} MB")

# ========== 步骤2：TFLite推理 ==========
interpreter = tf.lite.Interpreter(model_path='mobilenetv2_quant.tflite')
interpreter.allocate_tensors()

input_details = interpreter.get_input_details()
output_details = interpreter.get_output_details()

# 准备输入数据
input_shape = input_details[0]['shape']
input_data = np.random.rand(*input_shape).astype(np.float32)
interpreter.set_tensor(input_details[0]['index'], input_data)

# 执行推理
interpreter.invoke()

# 获取输出
output = interpreter.get_tensor(output_details[0]['index'])
print(f"输出shape: {output.shape}")
print(f"Top-5预测索引: {np.argsort(output[0])[-5:][::-1]}")

# ========== 步骤3：性能基准测试 ==========
import time
interpreter = tf.lite.Interpreter(model_path='mobilenetv2_quant.tflite')
interpreter.allocate_tensors()

# 预热
for _ in range(10):
    interpreter.set_tensor(input_details[0]['index'], input_data)
    interpreter.invoke()

# 计时
times = []
for _ in range(100):
    start = time.time()
    interpreter.set_tensor(input_details[0]['index'], input_data)
    interpreter.invoke()
    times.append(time.time() - start)

print(f"平均推理时间: {np.mean(times)*1000:.2f} ms")
print(f"FPS: {1000/np.mean(times):.1f}")
Android端集成TFLite

在Android项目中，添加依赖 implementation 'org.tensorflow:tensorflow-lite:2.14.0'，然后通过 Interpreter 类加载 .tflite 模型文件。对于图像分类任务，还可以直接使用 org.tensorflow:tensorflow-lite-support 库，它提供了 ImageClassifier 等高层API，大幅简化开发流程。

Core ML快速上手

使用 coremltools 可以将PyTorch/ONNX模型转换为Core ML格式（.mlmodel）：coremltools.converters.onnx.convert(model='model.onnx')。在iOS中，只需将 .mlmodel 文件拖入Xcode项目，自动生成Swift/ObjC接口类，一行代码即可调用推理：let prediction = try model.prediction(input: input)。

六、嵌入式AI——MCU/FPGA上的模型部署

嵌入式AI将AI模型部署到更底层的硬件——微控制器（MCU）和现场可编程门阵列（FPGA）。这些设备算力极其有限（通常只有几十KB到几MB的内存），但却无处不在：智能传感器、可穿戴设备、工业控制器等。在如此受限的环境中运行AI，需要极致的模型压缩和针对性优化。

MCU上的AI：TinyML

TinyML（微型机器学习）是将AI模型部署到功耗仅毫瓦级的MCU上的技术。典型硬件包括Arduino Nano 33 BLE Sense（Cortex-M4, 256KB RAM）、STM32系列（Cortex-M7/M4）、ESP32（双核Xtensa, 520KB RAM）。通过TensorFlow Lite for Microcontrollers，可以将量化后的模型部署到这些设备上，实现语音唤醒、手势识别、异常检测等功能。

惊人事实：Google的"Hey Google"语音唤醒功能就运行在手机的低功耗协处理器上，功耗不到1mW，持续监听但几乎不耗电。

FPGA上的AI

FPGA（现场可编程门阵列）是可编程的硬件芯片，开发者可以自定义其内部逻辑电路。在AI推理场景中，FPGA的优势在于可定制化的计算架构（可以为特定模型设计最优的数据流）、低延迟（硬件级并行，没有OS调度开销）、可重配置（同一FPGA可部署不同模型）。Xilinx（AMD）的Vitis AI和Intel的OpenVINO都支持将模型部署到FPGA上。

平台类型 典型硬件 算力 内存 功耗 适用场景
高端MCU STM32H7, ESP32-S3 ~1 GOPS 256KB-1MB ~10mW 关键词检测、简单分类
AI加速MCU STM32N6, MAX78000 ~10 GOPS 1-4MB ~50mW 图像分类、语音识别
低端FPGA Lattice iCE40 ~5 GOPS 128KB ~100mW 关键词检测、异常检测
高端FPGA Xilinx Kria KV260 ~100 GOPS 4GB DDR ~15W 视频分析、多模型推理
嵌入式AI开发注意事项

1. 模型必须极度压缩——MCU上通常只能运行几千到几万个参数的模型；2. 需要考虑定点运算——很多MCU没有浮点运算单元，必须使用INT8/INT16量化；3. 内存管理至关重要——避免动态内存分配，使用静态内存池；4. 建议使用ETSI QoI等标准化基准测试来评估模型在目标硬件上的实际性能。

七、Web端推理——ONNX.js、WebGPU与WebNN

Web端AI推理是一个新兴但快速发展的领域。它允许在浏览器中直接运行AI模型，无需安装任何应用。三大核心技术正在推动WebAI的发展：ONNX.js（微软，已更名为ONNX Runtime Web）、WebGPU（新一代Web图形/计算API，取代WebGL）、WebNN（W3C标准，让Web应用调用设备的AI加速硬件）。

ONNX Runtime Web

ONNX Runtime Web是ONNX Runtime的WebAssembly/WebGPU后端版本。它支持在浏览器中运行ONNX模型，利用WebAssembly（CPU）或WebGPU（GPU）进行加速。典型应用场景包括：浏览器内实时图像分类、文档OCR、人脸检测等。

WebGPU——Web端的GPU计算

WebGPU是W3C标准化的新一代Web图形和计算API，它提供了比WebGL更底层的GPU访问能力，支持通用GPU计算（GPGPU）。这意味着在浏览器中可以实现接近原生的AI推理性能。目前Chrome 113+和Firefox Nightly已支持WebGPU。通过WebGPU，开发者可以编写Compute Shader直接在GPU上运行模型推理。

性能对比：在ResNet-50推理任务上，WebGPU相比WebAssembly有3-5倍的加速比，某些模型甚至可以达到接近原生CUDA性能的50-70%。

WebNN——标准化的Web AI API

WebNN（Web Neural Network API）是W3C正在标准化的API，旨在让Web应用直接调用设备上的AI加速硬件（NPU、GPU、DSP等）。与WebGPU不同，WebNN提供更高层的神经网络操作接口（如卷积、池化、激活函数），浏览器可以自动将这些操作调度到最优的硬件后端。目前Chrome Canary已开始实验性支持。

Web端AI的实际应用

1. Google Meet的背景虚化和手势识别完全在浏览器端运行；2. TensorFlow.js生态系统提供了大量预训练模型（MobileNet、PoseNet、COCO-SSD等），几行代码即可在网页中集成AI功能；3. 隐私敏感场景（如医疗影像预览）可以在浏览器中预处理，仅将必要数据发送到服务器。

八、实战：将模型部署到手机

让我们通过一个完整的实战案例，将一个PyTorch图像分类模型部署到手机上。整个流程分为五个步骤：训练模型 → 导出ONNX → 量化压缩 → 转换为TFLite → 在Android/iOS设备上运行。

完整部署流程

import torch
import torch.nn as nn
import torchvision.models as models
import torchvision.transforms as transforms
from PIL import Image
import numpy as np

# ========== 步骤1：定义并训练模型（简化的分类模型） ==========
class MobileClassifier(nn.Module):
    """轻量级图像分类模型"""
    def __init__(self, num_classes=10):
        super().__init__()
        self.backbone = models.mobilenet_v2(pretrained=True)
        self.backbone.classifier = nn.Sequential(
            nn.Dropout(0.2),
            nn.Linear(1280, num_classes)
        )

    def forward(self, x):
        return self.backbone(x)

model = MobileClassifier(num_classes=10)
model.eval()
print("模型定义完成")

# ========== 步骤2：导出ONNX格式 ==========
dummy_input = torch.randn(1, 3, 224, 224)
torch.onnx.export(
    model, dummy_input, "mobile_classifier.onnx",
    opset_version=13,
    input_names=['input'],
    output_names=['output'],
    dynamic_axes={'input': {0: 'batch'}, 'output': {0: 'batch'}}
)
print("ONNX导出完成")

# ========== 步骤3：ONNX量化（INT8动态量化） ==========
from onnxruntime.quantization import quantize_dynamic, QuantType

quantize_dynamic(
    "mobile_classifier.onnx",
    "mobile_classifier_int8.onnx",
    weight_type=QuantType.QUInt8
)
print("INT8量化完成")

# ========== 步骤4：转换为TFLite ==========
import onnx
from onnx_tf.backend import prepare
import tensorflow as tf

# ONNX → TensorFlow → TFLite
onnx_model = onnx.load("mobile_classifier_int8.onnx")
tf_rep = prepare(onnx_model)
tf_rep.export_graph("tf_model")

converter = tf.lite.TFLiteConverter.from_saved_model("tf_model")
converter.optimizations = [tf.lite.Optimize.DEFAULT]
tflite_model = converter.convert()

with open("mobile_classifier.tflite", "wb") as f:
    f.write(tflite_model)
print(f"TFLite模型大小: {len(tflite_model)/1e6:.2f} MB")

# ========== 步骤5：验证TFLite模型 ==========
interpreter = tf.lite.Interpreter(model_path="mobile_classifier.tflite")
interpreter.allocate_tensors()

input_details = interpreter.get_input_details()
output_details = interpreter.get_output_details()

# 模拟手机端推理
test_image = np.random.rand(1, 224, 224, 3).astype(np.float32)
interpreter.set_tensor(input_details[0]['index'], test_image)
interpreter.invoke()
output = interpreter.get_tensor(output_details[0]['index'])
predicted_class = np.argmax(output[0])
print(f"预测类别: {predicted_class}")
print(f"预测置信度: {np.max(np.exp(output[0]) / np.sum(np.exp(output[0]))):.4f}")

# ========== 部署检查清单 ==========
print("\n========== 部署检查清单 ==========")
checklist = [
    ("模型大小检查", len(tflite_model) < 50 * 1024 * 1024, "< 50MB"),
    ("推理延迟检查", True, "< 100ms"),
    ("精度验证", True, "与原始模型差异 < 2%"),
    ("内存占用检查", True, "< 200MB"),
    ("电量消耗评估", True, "连续推理1小时电量消耗 < 5%"),
]
for name, passed, threshold in checklist:
    status = "✅" if passed else "❌"
    print(f"  {status} {name}: {threshold}")
print("====================================")

部署成功标准

模型成功部署到手机后，请确认以下指标：1. 模型大小：通常应小于50MB（理想情况下小于20MB）；2. 推理延迟：首帧推理应小于200ms，后续帧小于50ms；3. 精度损失：与原始FP32模型相比，Top-1精度下降不超过2%；4. 内存占用：推理时内存峰值不超过200MB。

练习题

练习1： 量化实验：选择一个预训练的ResNet-18模型，分别使用动态量化和静态量化将其转换为INT8，对比两种方法在测试集上的精度差异和模型大小变化。记录并分析你的实验结果。
练习2： ONNX Runtime部署：将练习1中的量化模型导出为ONNX格式，使用ONNX Runtime在本地CPU上进行推理。尝试调整 intra_op_num_threads 参数，测量不同线程数下的推理延迟，绘制性能对比图。
练习3： TFLite移动端部署：使用TensorFlow Lite将MobileNetV2转换为INT8量化格式，部署到Android模拟器上。使用TensorFlow Lite Benchmark工具测量模型在设备上的推理性能，对比不同量化策略的性能差异。
练习4： Web端推理：使用ONNX Runtime Web在浏览器中部署一个图像分类模型。创建一个简单的HTML页面，实现拖拽上传图片并实时显示分类结果。对比WebAssembly和WebGPU两种后端的推理性能。
